LLM Builder Best Practices

A set of standing rules for any LLM-driven build session. File structure, documentation discipline, decision logging, wrap rituals. One file you drop into your AI once, and it documents every project from then on.

What this is

Builds get messy fast. Decisions get lost. Stacks drift. Next Monday you reopen the project and your AI has no idea what it built. These rules keep your project legible to you, to other LLMs, and to your future self.

First-time workspace setup (run once)

Before any project, set up ONE master "work" folder on your computer. Every project lives inside it as a subfolder. At the ROOT of that master folder, three shared files protect you and save you from copying API keys around.

FilePurpose
.gitignoreTells git which files to never upload to GitHub. Protects secrets, junk, dependency folders.
.envYour master secrets file. API keys, tokens, passwords. ONE file, shared across every project. Never committed.
.env.exampleThe safe, shareable version. Lists key names with placeholder values. Safe to commit.

Why one shared .env at the root: put your API keys in once, every project below reads them. No copying keys between projects. Add a new key once, every project gets it.

Paste this into your LLM the first time you set up your master work folder:

I'm new to coding. I want to set up ONE master folder on my computer that holds every future project as a subfolder. I want a single .env file at the TOP of that folder that every project inside can share, so I never have to copy API keys between projects. Please do the following: 1. Confirm location. Ask me what folder I want to use as my master "work" folder. Confirm the full path before creating anything. Everything below goes at the ROOT of that folder, NOT inside a subproject. 2. Create .gitignore at the root. Include sections with short plain-English comments: - Secrets: .env, .env.local, *.key, *.pem, credentials.json, token.json - OS junk: .DS_Store, Thumbs.db - Editor folders: .vscode/, .idea/ - Dependency folders: node_modules/, venv/, __pycache__/ - Logs: *.log - Build output: dist/, build/ 3. Create .env at the root. Empty except for a comment header: "# Master secrets file. Shared across every project in this folder. NEVER commit or share this file." 4. Create .env.example at the root. Comment header explaining: "This file shows which keys exist. Copy this to .env and fill in real values." Include one example line: OPENAI_API_KEY=your_key_here 5. Explain to me, plainly: - Why I have ONE .env at the root instead of one per project - How a project inside this folder reads the root .env - What to do when I add a new API key later (real value to root .env, placeholder name to root .env.example) - The golden rule: if a file holds a real secret, it belongs in .gitignore. If it's a template showing key names only, it does not. 6. Do NOT run any git commands. Just create the three files at the root and explain.

Run this once. Every new project you start inside the folder inherits the shared .env automatically.

The four documents your AI maintains

WORKBENCH.md
What just happened
Per-session log. 100-300 words. Newest at top. Decisions tagged with reasoning.
ARCHITECTURE.md
How the thing is built
Stack, file map, data flow, key decisions. Updated when the stack changes.
USER_GUIDE.md
What the user gets
Plain language. What it does, how to use it, settings. Updated on any user-facing change.
TODOS.md
What's next
Next / Soon / Someday. One line per item. No priorities, no labels, no dates.

The file also installs honesty rules (no inventing file paths, no fake success claims), a clean communication style (lead with the answer, no preamble, no em dashes), and an automatic wrap-up routine that fires when you end a session.

How to use it

  1. Download llm-builder-bps.md. One file. That's all you need.
  2. Drop it into your LLM chat. Claude, Codex, ChatGPT, any of them. Paste the contents or upload the file. Whichever your tool prefers.
  3. Tell it to install. Install this. Apply it to every session going forward. It will acknowledge once, then operate by these rules silently from that point on.

Why each rule exists

  • Workbench. Captures the decisions and reasoning that git log doesn't.
  • Architecture. Stops you re-explaining your stack to your AI every Monday.
  • User Guide. Keeps you honest about what the thing actually does for the person using it.
  • TODOs. Stops you from carrying unfinished work in your head.

Together the four close the loop. Your AI builds, your AI documents, you trust the docs, the next session starts warm.
