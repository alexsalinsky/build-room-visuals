LLM Builder Best Practices

A set of standing rules for any LLM-driven build session. File structure, documentation discipline, decision logging, wrap rituals. One file you drop into your AI once, and it documents every project from then on.

What this is

Builds get messy fast. Decisions get lost. Stacks drift. Next Monday you reopen the project and your AI has no idea what it built. These rules keep your project legible to you, to other LLMs, and to your future self.

The four documents your AI maintains

WORKBENCH.md
What just happened
Per-session log. 100-300 words. Newest at top. Decisions tagged with reasoning.
ARCHITECTURE.md
How the thing is built
Stack, file map, data flow, key decisions. Updated when the stack changes.
USER_GUIDE.md
What the user gets
Plain language. What it does, how to use it, settings. Updated on any user-facing change.
TODOS.md
What's next
Next / Soon / Someday. One line per item. No priorities, no labels, no dates.

The file also installs honesty rules (no inventing file paths, no fake success claims), a clean communication style (lead with the answer, no preamble, no em dashes), and an automatic wrap-up routine that fires when you end a session.

How to use it

  1. Download llm-builder-bps.md. One file. That's all you need.
  2. Drop it into your LLM chat. Claude, Codex, ChatGPT, any of them. Paste the contents or upload the file. Whichever your tool prefers.
  3. Tell it to install. Install this. Apply it to every session going forward. It will acknowledge once, then operate by these rules silently from that point on.

Why each rule exists

  • Workbench. Captures the decisions and reasoning that git log doesn't.
  • Architecture. Stops you re-explaining your stack to your AI every Monday.
  • User Guide. Keeps you honest about what the thing actually does for the person using it.
  • TODOs. Stops you from carrying unfinished work in your head.

Together the four close the loop. Your AI builds, your AI documents, you trust the docs, the next session starts warm.
