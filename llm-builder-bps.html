LLM Builder Best Practices

A set of standing rules for any LLM-driven build session. File structure, documentation discipline, decision logging, wrap rituals. One file you drop into your AI once, and it documents every project from then on.

What this is

Builds get messy fast. Decisions get lost. Stacks drift. Next Monday you reopen the project and your AI has no idea what it built. These rules keep your project legible to you, to other LLMs, and to your future self.

Copy-paste install (one shot)

Paste this entire block into a fresh chat with Claude, ChatGPT, Codex, Cursor, or any LLM. One paste does everything: installs the best practices as standing rules across the conversation, sets up your master "work" folder, and creates a workspace index your AI will read at the start of every future session.

Step 3, install these standing rules and operate by them silently from now on: THE FOUR LIVING DOCUMENTS — every project I work on maintains four files alongside the code, and you keep them current as we work: - WORKBENCH.md = per-session log. Append-only, newest at top. 100 to 300 words per entry. Tag decisions with [DECISION] and a reason. No code diffs. - ARCHITECTURE.md = stack, file map, data flow, key decisions. Update when the stack changes. - USER_GUIDE.md = plain-language doc for the person who uses the thing. Update on any user-facing change. - TODOS.md = honest list of what's next, grouped Next / Soon / Someday. One line per item, leading verb. If any of the four is missing when we start work in a project, create it with a clean skeleton. WORKBENCH ENTRY FORMAT (prepend to top of WORKBENCH.md after any session that modifies files or makes a decision): ## YYYY-MM-DD HH:MM, [tool used] **Focus:** one-line summary. **Next session, start here:** Read / In flight / Single next move. **What happened:** bullets. **Decisions:** [DECISION] chose X over Y because Z. ("None" if no decisions.) **Still open:** items needing follow-up. **Files touched:** comma-separated. --- WORKSPACE NAVIGATION RULE — at the start of every future session, read root WORKSPACE.md first. When I mention a project by name or ask anything cross-project, use WORKSPACE.md to find the right project folder, then read the top entry of that project's WORKBENCH.md before answering. When we create a new project subfolder, add a row to WORKSPACE.md the same session. HONESTY RULES — Don't claim success for something you didn't verify. Don't invent file paths, function names, or URLs you haven't seen. Don't add error handling for cases that can't happen. Don't write comments that restate what the code does, only the non-obvious why. COMMUNICATION STYLE — Lead with the answer. Bullets and tables over paragraphs. Short over long. No preamble. The four documents your AI maintains

WORKBENCH.md
What just happened
Per-session log. 100-300 words. Newest at top. Decisions tagged with reasoning.
ARCHITECTURE.md
How the thing is built
Stack, file map, data flow, key decisions. Updated when the stack changes.
USER_GUIDE.md
What the user gets
Plain language. What it does, how to use it, settings. Updated on any user-facing change.
TODOS.md
What's next
Next / Soon / Someday. One line per item. No priorities, no labels, no dates.

The file also installs honesty rules (no inventing file paths, no fake success claims), a clean communication style (lead with the answer, no preamble, no em dashes), and an automatic wrap-up routine that fires when you end a session.

How to use it

  1. Download llm-builder-bps.md. One file. That's all you need.
  2. Drop it into your LLM chat. Claude, Codex, ChatGPT, any of them. Paste the contents or upload the file. Whichever your tool prefers.
  3. Tell it to install. Install this. Apply it to every session going forward. It will acknowledge once, then operate by these rules silently from that point on.

Why each rule exists

  • Workbench. Captures the decisions and reasoning that git log doesn't.
  • Architecture. Stops you re-explaining your stack to your AI every Monday.
  • User Guide. Keeps you honest about what the thing actually does for the person using it.
  • TODOs. Stops you from carrying unfinished work in your head.

Together the four close the loop. Your AI builds, your AI documents, you trust the docs, the next session starts warm.
