README + handoffThe README is the demo before the demo.

You've built a thing. It runs on your machine. You know which file to edit, which command to run, where the API key lives. None of that is in your project yet. It's only in your head.

A publisher writes it down. Not because someone is reading it. Because future you, three weeks from now, won't remember either.

The handoff test

5minutes

Can a teammate clone your project, read your README, and get it running in five minutes — without asking you a question?

Yes = you shipped a publisher-grade artifact. No = you built a thing that only works when you're in the room.

The 7 fields a Build Room README needs

Not more. Not less. Each one earns its line.

1

One-line description

What this thing does, in 15 words or fewer. No background, no story. Just the job.

# Pull last month's PayPal payments and flag overdue clients.
2

Run it

The exact command. Not "how to run." The command itself, copy-pasteable.

npm run report
3

What you need first

Deps, accounts, env vars. Anything the command depends on that isn't in the repo.

Node 20+ · PayPal sandbox key · .env file (see .env.example)
4

Where the secrets live

Name the env vars. Show .env.example. Say where the real values come from. Do NOT paste real keys.

PAYPAL_API_KEY in .env (get from PayPal dashboard → Apps → MyApp)
5

What it produces

Where the output goes. A file? A Slack message? A row in a sheet? Name it.

Writes ./out/overdue-YYYY-MM-DD.csv
6

Known limitations

The honest list. Things you'd warn a teammate about. Two or three lines.

- Only handles USD
- Skips refunds > 90 days old
- One PayPal account at a time
7

When it breaks

The first thing to check. The error message you've seen twice. The fix you keep forgetting.

If 401: regenerate the PayPal key, it expires every 30 days.

Sample README, end to end

# Overdue Client Pinger Pulls last month's PayPal payments and flags clients who haven't paid this month. ## Run it ``` npm install cp .env.example .env # fill in PAYPAL_API_KEY npm run report ``` ## What you need - Node 20+ - PayPal sandbox or live API key - Stripe account is optional (Stripe fallback only) ## Secrets - PAYPAL_API_KEY in .env (get from PayPal dashboard → Apps → MyApp) - NEVER commit .env. .gitignore already excludes it. ## Output Writes ./out/overdue-YYYY-MM-DD.csv. Columns: client, last_paid, days_overdue, email. ## Known limits - USD only - Skips refunds > 90 days - One PayPal account at a time ## When it breaks - 401: PayPal key expired, regenerate in dashboard - empty CSV: month boundary, check the date arg

The pattern: write it WITH Claude, not after

The bad pattern: finish the build, then sit down to write the README. You forget half the gotchas. The README is generic. The handoff test fails.

The Build Room pattern: when you start the build, ask the AI to draft the README first. Each time you add a thing that needs setup, ask the AI to update the README at the same time. By the time the build is done, the README is done.

Prompt to keep handy:

"Look at what changed in this session. Update the README so a new teammate could run this in 5 min without asking me anything. Keep the 7-section structure."

Demo Day connection

S4 is one week away. Your README is the script for the first 30 seconds of your demo:

  • "Here's what it does" — your one-line description
  • "Here's how to run it" — your run-it command, shown live
  • "Here's what it produces" — show the output file or message

If you can't open your README and pull those three lines without rewriting them, the demo is the wrong shape too.

The README is the demo before the demo. Write it now, demo it later.
