How to Store Your Credentials

API keys and tokens are passwords. This page is the floor for handling them so you don't leak one in week one. Read it once, follow the pattern every time.

One .env file at the root of your project. Every key lives there. Add .env to .gitignore before your first commit. Never paste a key into chat.

1. The Pattern

Inside your project folder, create one plain text file named .env (note the leading dot). Every API key and token your build uses goes in this file. One per line. Format:

KEY_NAME=value

No quotes unless the value itself contains spaces. No spaces around the equals sign. One key per line. Comments start with #.

Your AI assistant, your scripts, and any tool you build all read keys from this one file. When a key changes, you change it in one place.

2. Hide it from GitHub

The same project folder needs a second file named .gitignore. Add this line to it before you ever commit:

.env

This tells Git to never track, never upload, never sync your .env file. Without this line, the first time you push to GitHub, your keys go public. Bots scan public GitHub for new keys constantly. Leaks get exploited in minutes, not days. GitGuardian publishes an annual report on this if you want the gory details: State of Secrets Sprawl.

Add the .gitignore line first. Then create the .env. Then start adding keys. The order matters.

3. Edit it yourself, not through the LLM

Open .env in a plain text editor (VS Code, Notepad, TextEdit). Paste the key in by hand. Save.

Do not ask your AI assistant to add the key for you. The moment you paste a key into a chat, it's in that chat's transcript, possibly in logs, possibly in training pipelines depending on the tool, and definitely visible on your screen during any screen-share.

Instead, once the key is in the file, tell the assistant where to find it:

# What you say to the LLM:
The key is in .env as OPENAI_API_KEY. Read it from there.

The assistant reads the file from disk, uses the key, and never has to see it printed in the chat.

4. Do and Don't

Do
  • Keep one .env per project, at the project root
  • Add .env to .gitignore before your first commit
  • Edit .env in a text editor yourself
  • Reference keys by name when talking to your AI assistant
  • Rotate any key you suspect was seen by anyone
Don't
  • Paste a key into Skool, Slack, Discord, or any chat thread
  • Screenshot a key, even partially
  • Screen-share with .env open and no redaction
  • Commit .env to GitHub, ever, even once
  • Email a key to a teammate (use a password manager)

5. If You Leak One

Assume the key is compromised the moment it lands anywhere public. Deleting the commit, deleting the message, deleting the screenshot is not enough. Bots and caches see it before you can pull it back.

The fix is to rotate. Go to the provider, revoke the old key, generate a new one, paste it into .env. The leaked one is now useless.

Rotation order: issue the new key first, swap it in .env, confirm your build still works, then revoke the old one. This keeps your build running through the swap.

6. A Worked Example

A minimal project folder looks like this. Three files, all at the root.

.gitignore

.env
node_modules/
__pycache__/

.env

# API keys for this project. Never commit this file.
OPENAI_API_KEY=sk-proj-XXXXXXXXXXXXXXXXXXXXXXXX
ANTHROPIC_API_KEY=sk-ant-XXXXXXXXXXXXXXXXXXXXXXXX
GITHUB_TOKEN=ghp_XXXXXXXXXXXXXXXXXXXXXXXX

How code reads it

Most languages have a one-line loader. Python uses python-dotenv. Node uses dotenv. The pattern is the same everywhere: load the file, then read keys by name from the environment. You don't paste the value into your code, you reference the name.

# Python example
from dotenv import load_dotenv
import os

load_dotenv()
key = os.getenv("OPENAI_API_KEY")

If your AI assistant is doing the building, just tell it which key from .env the task needs. It handles the rest.
