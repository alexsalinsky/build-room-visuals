Use your favorite tool, default to Codex if you don't have one. The discipline transfers anywhere: Claude Code, Cursor, ChatGPT desktop, Codex CLI.

1. .gitignore

The first file you create in every new project. It tells Git what to skip when you commit. Without it, your secrets, your build artifacts, and your operating system's invisible files all end up on GitHub.

Paste this as your starter:

# Secrets, never commit
.env
.env.local
.env.*.local

# Dependencies
node_modules/
__pycache__/
*.pyc
.venv/
venv/

# Build output
dist/
build/
.next/
out/

# Editor + OS junk
.DS_Store
Thumbs.db
.vscode/
.idea/

# Logs
*.log
npm-debug.log*

The first three lines are the load-bearing ones. The rest just keeps your repo clean. Add lines as you discover new things to ignore. Never remove the secrets block.

2. .env + credentials hygiene

Plain text file at the project root. Every API key, token, and secret lives here, one per line. Your scripts read keys from this file. Your AI tool reads keys from this file. Change a key once, in one place.

# Format: no quotes, no spaces around =, one per line
OPENAI_API_KEY=sk-proj-XXXXXXXXXXXXXXXXXXXXXXXX
ANTHROPIC_API_KEY=sk-ant-XXXXXXXXXXXXXXXXXXXXXXXX
GITHUB_TOKEN=ghp_XXXXXXXXXXXXXXXXXXXXXXXX

Ship a .env.example alongside it

Same keys, no values. This is the one you DO commit. It tells anyone who clones your project which variables they need to fill in.

# Copy this file to .env and fill in real values
OPENAI_API_KEY=
ANTHROPIC_API_KEY=
GITHUB_TOKEN=

The rules

  • Edit .env in a text editor by hand. Never paste a key into a chat thread.
  • Reference keys by name when you talk to your tool: "The key is in .env as OPENAI_API_KEY, read it from there."
  • If you ever leak a key, rotate it at the provider. Deleting the commit is not enough, bots see public repos in minutes.
  • One .env per project, at the project root. Not in a subfolder, not in your home directory.
Order matters. Create .gitignore first. Then create .env. Then add keys. If you create .env first and forget to ignore it, the first git add . stages your keys.

3. The rule-pack

Six markdown files you drop into your tool's config. They make your AI document itself as it builds, keep decisions out of chat threads, and let next Monday's session start warm instead of cold.

This is the same stack Alex's own workspace runs on. Distilled into something you can install in two minutes.

WORKBENCH.md
What just happened
Per-session log. 100-300 words. Newest at top. Decisions tagged with reasoning.
ARCHITECTURE.md
How the thing is built
Stack, file map, data flow, key decisions. Updated when the stack changes.
USER_GUIDE.md
What the user gets
Plain language. What it does, how to use it, settings. Updated on any user-facing change.
TODOS.md
What's next
Next / Soon / Someday. One line per item. No priorities, no labels, no dates.

The pack also installs a handoff template (for cold-start session transfer) and a wrap protocol (fires automatically when you end a session). Plus honesty rules: no inventing file paths, no false success claims, no em dashes.

4. Wire it in

One file. Drop it into your tool once. It runs by these rules silently from that point on.

llm-builder-bps.md ↓
  1. Download llm-builder-bps.md above. One file. That's the whole rule-pack.
  2. Open your project folder in your coding tool. Codex, Claude Code, Cursor, anything that reads project files.
  3. Drop the file in the project root and tell your tool to install it. In Codex or Claude Code: paste the contents into chat and say "Install this. Apply to every session going forward." The tool stores it as project memory (Codex: AGENTS.md; Claude: CLAUDE.md).
  4. Verify by asking: "What rules are you operating by?" It should list the four docs, the wrap protocol, and the honesty rules. If it doesn't, paste again.

From here on, the tool maintains the four documents as you build. You'll see WORKBENCH.md, ARCHITECTURE.md, USER_GUIDE.md, and TODOS.md appear in your project folder. When you end a session, the wrap protocol fires and your log gets a fresh entry.

The whole setup in one glance

my-project/
├── .gitignore          # created first, hides .env
├── .env                # your keys, never committed
├── .env.example        # template, safe to commit
├── AGENTS.md           # (Codex) installed rule-pack
├── CLAUDE.md           # (Claude Code) installed rule-pack
├── WORKBENCH.md        # tool maintains this
├── ARCHITECTURE.md     # tool maintains this
├── USER_GUIDE.md       # tool maintains this
├── TODOS.md            # tool maintains this
├── README.md
└── src/
    └── main.py

That's the standard. Same shape in every Build Room project. Pick it up, set it up once, build with confidence.
