Tokenomics

Efficiency and ethics are the same thing.
A mental model for getting more out of LLMs without burning your limits, your wallet, or the planet.
Why this matters

The hard ceiling.

The first reason to care about tokens isn't money. It's the wall. If you're not efficient, you hit the token limit and just can't develop anymore. Paying more pushes the wall back. It doesn't remove it. Context keeps growing.

Mid-build

Usage limit reached

Daily allowance
100% used. Resets in 4h 18m.
You've hit your limit.
Continue with a smaller model or wait for reset. Upgrade for more.

The wall always exists. The number isn't the point. Efficiency is the point.

Every message re-reads the full thread from scratch. A 100k-token thread costs 100k tokens before you type a word. That's how you hit the wall faster than you expected.

When efficiency starts mattering

For yourself? Don't bother. For users? Bother.

If you're running a job 10 times a month for yourself, 30 cents a run does not matter. Don't optimize. Build it.

Personal scale

You. Ten runs a month.

30¢ per run · $3/month total
It's coffee money. Move on.

Build the thing. Get the output. Don't waste a day shaving pennies.
Verdict: don't optimize.
Publisher scale

1,000 users. Daily.

30¢/run × 30,000 runs/month = $9,000/month
Same code. 10× cheaper version: $900/month.

Plus the slowdown: bloated context means slower responses for everyone.
Verdict: efficiency is the product.
At scale, every wasted token is wasted processing time, wasted server time, wasted energy. And it slows down your results.
The reframe

Ethics IS efficiency.

Every token isn't free. Tokens are compute, compute is power, power is heat. Heat means cooling. Cooling means water. Wasted tokens have a real-world footprint.

🔤Wasted tokens
→
⚡Wasted compute
→
💧Wasted water + power
This is the Build Room ethics frame. Don't burn what you don't have to. It's the same discipline whether you call it efficiency, ethics, or just good engineering.
How to be efficient

Three knobs you can turn.

You don't need to optimize anything until you ship. But you DO need to know what dials exist, so you can reach for them when scale hits.

KNOB 1

Right model for the job

Match the model size to the task. Small task, small model. Don't reach for the biggest hammer to drive a tack.

KNOB 2

From "LLM-built" to "LLM-builds-once"

The biggest win. Use the LLM to figure out the logic. Then have the LLM write code that runs without calling an LLM next time.

KNOB 3

Manage threads + context

Don't ship the whole world into context every run. Chunk it. Burn threads when they bloat. Brief replaces archive.

Knob 1

Right model. Right job.

Bigger models think harder. They also cost more, take longer, and burn more. If a small model can do the job, use the small model.

Small / cheap / fast
Haiku
Sorting, classifying, tagging, simple Q&A, structured extraction, "is this X or Y?"
Mid / balanced
Sonnet
Writing, summarizing, code edits, planning, most conversational work.
Large / heavy
Opus
Hard reasoning, complex architecture, multi-step agent work, anything that fails on smaller models.
Default to mid. Drop down for repetitive sorting jobs. Reach up only when the smaller model fails. "Big model on everything" is the most expensive bad habit in AI.
Knob 2 · The biggest knob

Let the LLM write the loop. Then exit the loop.

Most builds don't need an LLM in the loop forever. The LLM is the architect. Once you understand how something works, it becomes plain code that runs without paying for tokens at all.

Phase 1 · build it WITH the LLM

LLM is in the loop on every run

You're figuring it out. The LLM helps you think, drafts the logic, tries options. Every run costs tokens. That's fine. You're learning the shape.

→
Phase 2 · ship it WITHOUT the LLM

LLM wrote a script that runs deterministic

The logic is locked. The LLM wrote you a function, a cron job, a script. It runs by itself. Zero tokens. Instant. Free.

Build Room rule: if a task is repeating, the LLM's job is to write the thing that does the task, not to keep doing the task.
Knob 3

Don't ship the world. Ship the brief.

Every message in an LLM thread re-reads the entire thread. A bloated thread is wasted tokens on every exchange. Briefs and project files solve this.

  • Burn threads when they bloat. Start fresh. Paste a one-page brief instead of scrolling back.
  • Project briefs over full repos. Point Claude at PROJECT.md, not 200 source files. Same knowledge, fraction of the tokens.
  • Machine-first formats. Markdown over rich text. APIs over screenshots. Every UI an LLM has to navigate is tokens spent on overhead, not work.
Already covered in S2

This is "threads burn, files remain" read at the cost level. The discipline you learned in S2 isn't just about not losing context. It's also the cheapest single thing you can do to keep your tokens going to work, not overhead.

A separate concern

When the data shouldn't leave.

Efficiency-as-ethics is about waste. There's a different ethical question that lives next to it: some data shouldn't go through a public model at all.

When you reach for a private LLM

Sensitive data, regulated industries, client confidentiality

Health records. Legal discovery. Financial documents you're not the owner of. Proprietary code under NDA. Anything where the question "did the data owner agree to this being read by a public LLM?" is "no" or "I don't know."

Options: self-hosted models (Llama / Mistral on your own server), enterprise LLM contracts with no-training guarantees, on-prem deployments. Slower, more setup, sometimes less capable. But the data stays where it belongs.

Sometimes the right answer is "use no LLM at all for this part." That's also valid.
This is one slide because for most Build Room projects, it doesn't apply. But when it applies, it's the whole game.
The mindset shift

At publisher tier, every token has a cost.

Personal scale: build it however. Publisher scale: every token is money, time, electricity, water. Tokenomics isn't optimization theater. It's the discipline of not wasting what you don't have to.
